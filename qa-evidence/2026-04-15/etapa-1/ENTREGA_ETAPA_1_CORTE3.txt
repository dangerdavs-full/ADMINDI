Etapa exacta
-----------
Etapa 1, corte 3 (trazabilidad de evidencia + alineación inbox ↔ backend approve/reject).

Implementado en código
----------------------
1. **ActionTaskController** — Antes de delegar a PropertyService:
   - 404 si la tarea no existe.
   - 403 si `task.userId` no coincide con el usuario autenticado.
   - 409 si la tarea no está `OPEN` (evita doble aprobación/rechazo por esta ruta).
   - 422 (UNPROCESSABLE_ENTITY) si `eventType` ≠ `PROPERTY_DELETE_REQUESTED`, con mensaje que incluye el tipo real.
   Constante pública: `EVENT_PROPERTY_DELETE_REQUESTED`.

2. **PropertyService** — `approveDeleteWithReauth`:
   - Valida estado OPEN, tipo, titular (`ownerId`) y arrendamiento activo **antes** de `verifyReauth` (no se pide contraseña si la operación va a fallar por reglas de negocio).
   - Arrendamiento activo: `422 UNPROCESSABLE_ENTITY` con mensaje explícito (antes era 400 genérico vía RuntimeException).
   - Rechazo (`rejectDeleteProperty`): mismas comprobaciones de OPEN + tipo + titular.

3. **GlobalExceptionHandler** — Manejador dedicado para `ResponseStatusException` (status HTTP fiel; cuerpo `{"message":"..."}`).

4. **Frontend ActionTaskInbox** — Botones «Aprobar eliminación» / «Rechazar solicitud» y modal `ReauthDeleteModal` **solo** si `eventType === PROPERTY_DELETE_REQUESTED` y `OPEN`. Otras tareas OPEN muestran texto de gestión en módulo correspondiente (sin CTA que llame a approve genérico).

5. **actionTaskService** — Export `EVENT_PROPERTY_DELETE_REQUESTED` alineado con backend.

6. **Pruebas** — `ActionTaskControllerApproveContractTest` (MockMvc standalone + Mockito):
   - approve + tipo incorrecto → 422 y no invoca PropertyService.
   - approve + RESOLVED → 409.
   - approve + flujo válido → 200 y delegación verificada.
   - reject + tipo incorrecto → 422 y no invoca PropertyService.

7. **Maven Surefire** — `argLine=-Dnet.bytebuddy.experimental=true` para ejecutar Mockito en JDK 25 del entorno de desarrollo.

Causa exacta del 400 observado (antes de este corte)
----------------------------------------------------
- El **frontend** mostraba «Aprobar» para toda tarea `OPEN`.
- El **backend** `POST /api/tasks/{id}/approve` delegaba siempre en `PropertyService.approveDeleteWithReauth`, que exigía `eventType == PROPERTY_DELETE_REQUESTED` y titular; si no, lanzaba `RuntimeException("Operación no autorizada.")`.
- **GlobalExceptionHandler** mapeaba cualquier `RuntimeException` a **HTTP 400** con `{"message":"Operación no autorizada."}` — mensaje ambiguo para una causa concreta (desajuste de tipo de tarea).

Fix exacto
----------
- Alineación UI: solo flujo de eliminación con reauth en bandeja para `PROPERTY_DELETE_REQUESTED`.
- Backend: validación explícita en controlador + códigos **409** / **422** / **403** / **404** vía `ResponseStatusException` y handler dedicado; reauth solo tras reglas de negocio válidas en `approveDeleteWithReauth`.
