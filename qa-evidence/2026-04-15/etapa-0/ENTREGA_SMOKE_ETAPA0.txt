-------
- Cadena **tenant multi-contexto** (login → contexts → select-context → switch-context → refresh → reuse refresh → logout → probe): **verde** (`SMOKE_RESULT=PASSED_AUTH` en SMOKE_RESULT.txt).

Backend

Variables usadas (tenant multi + contexto real)
------------------------------------------------
- SMOKE_EMAIL / SMOKE_PASSWORD: usuario tenant multi QA (`qa.etapa0.tenant.multi@test.local` + contraseña común documentada en QA_SEED_USERS.txt).
- SMOKE_CONTEXT_ID: **id devuelto por la API** para “QA Owner Alpha” en el mapa `organizations` del login (en esta BD fue `760b2ff2-2fb6-4bd5-a441-4c8c1aa2b1a0`, no el UUID fijo histórico de la semilla). Para corridas futuras: copiar desde el JSON de login o desde GET /auth/contexts.

Owner Gmail (archive)
---------------------
- Tras POST /auth/login, la cuenta owner indicada devuelve **MFA challenge** (`mfaChallengeToken` presente). El token previo a MFA no autoriza GET /api/tenants → 403 y no se puede resolver `tenantProfileId` ni ejecutar DELETE.
- **Se requiere un código TOTP actual** del usuario. El script ahora admite `SMOKE_ARCHIVE_MFA_CODE` (y POST /auth/mfa/verify para el owner archive) además de `SMOKE_OWNER_TOKEN_BEARER` ya verificado.
- El inquilino del escenario archive (`SMOKE_TENANT_EMAIL`) debe ser uno que **aparezca en GET /tenants** para ese owner tras MFA. Si solo existe semilla QA, el dueño de ese expediente es el owner QA “archive”, no necesariamente el owner Gmail salvo que la BD lo tenga así.
