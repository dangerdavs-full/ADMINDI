Resumen
Implementar ADMINDI — Administración Inteligente de Inmuebles como proyecto greenfield con Java 21 + Spring Boot 3, React + TypeScript + Vite, PostgreSQL, Redis y almacenamiento S3-compatible.
Tomar tu especificación nueva como fuente principal del producto; el plan anterior queda reemplazado.
Alcance v1: SUPER_ADMIN, OWNER, PROPERTY_ADMIN, ACCOUNTANT, TENANT; REAL_ESTATE_AGENT y MAINTENANCE_PROVIDER quedan modelados desde ahora, pero sus portales completos pasan a fase 2.
Integraciones v1: Mercado Pago real; WhatsApp y n8n mediante adapters/eventos y feature flags, priorizando primero los flujos web end-to-end.
Usar admindi también como naming técnico base: branding, slug interno, issuer JWT, nombres de app y configuración.
Cambios de Implementación
1. Seguridad, contexto y permisos
Autenticación con JWT RS256, refresh rotatorio, revocación por jti en Redis, MFA obligatorio para SUPER_ADMIN, OWNER, PROPERTY_ADMIN y ACCOUNTANT.
Modelo de acceso por User + OwnerMembership + PermissionTemplate + PermissionGrant; PROPERTY_ADMIN y ACCOUNTANT soportan múltiples contextos de dueño.
El ACCOUNTANT será multi-owner con selector de contexto igual al admin; dentro de cada owner ve todo el portafolio de ese dueño, no un subconjunto de inmuebles.
RLS en PostgreSQL para todos los recursos con owner_id; SUPER_ADMIN usa bypass administrativo controlado.
Reautenticación obligatoria para permisos destructivos, aprobación de convenios, borrados sensibles, cambios de permisos y exportaciones sensibles; todo queda auditado.
2. Dominio y módulos v1
Núcleo: Owner, User, Property, Unit, Tenant, Lease, PhotoAsset, LeaseDocument.
Cobranza: Payment, PaymentAllocation, MonthlyLedger, PaymentAgreement, ChargeRule, LateFeeRule, MonthlySummary.
Operación: MaintenanceContact, MaintenanceTicket, MaintenanceBudget, ExpenseCommitment, Vacancy, VacateRequest.
Plataforma: AuditEvent, ActionToken, WebhookReceipt, OutboundEvent, RefreshTokenSession.
Los contactos de mantenimiento y agentes inmobiliarios se modelan desde v1 para permitir asignaciones, historial y futura expansión de portales.
3. Resumen mensual contable
El MonthlySummary de v1 se define con base caja + compromisos: ingresos cobrados, ingresos pendientes, egresos pagados, egresos comprometidos, neto del mes, cuentas por cobrar y compromisos sin pagar.
El resumen debe servir tanto al dueño como al contador, así que incluirá desglose por inmueble y por unidad, referencias de pago, método de pago, fechas, saldos, convenios activos y estatus de cobro.
Se generarán dos salidas del mismo dataset:
PDF: vista ejecutiva y legible para revisión mensual.
Excel: vista contable operable con hojas separadas para Resumen, Ingresos, Cobranza pendiente, Egresos, Convenios y Facturación.
El ACCOUNTANT y el OWNER podrán descargar PDF y Excel del resumen mensual; PROPERTY_ADMIN solo si tiene REPORT_EXPORT.
El reporte de facturación sigue separado, pero conectado al resumen, con columnas suficientes para CFDI y trabajo contable mensual.
4. APIs e interfaces públicas
Mantener en v1: /auth/*, /admin/*, /owners/*, /permissions/*, /properties/*, /units/*, /leases/*, /tenants/*, /payments/*, /agreements/*, /maintenance/*, /vacancies/*, /reports/*, /files/*, /integrations/mercadopago/*, /integrations/whatsapp/webhook, /integrations/n8n/events.
El JWT final conservará sub, type, ownerId, role, permissions, providerType, jti, iat, exp; BASE solo sirve para selección de contexto.
Tipos públicos mínimos: MaintenanceFlowMode, MaintenanceCompensationModel, ExpenseStatus, PaymentStatus, AgreementStatus, LedgerStatus, UnitOccupancyStatus, VacancyStatus.
Añadir contratos de exportación para reportes con parámetros por owner, mes, año, propertyId opcional, formato=pdf|xlsx.
Usar naming técnico admindi en issuer JWT, claves de configuración, namespaces de backend/frontend y documentación base.
5. Entrega por fases
Fase 0: bootstrap del monorepo, CI, migraciones, auth base, RLS, auditoría y layout frontend.
Fase 1: SUPER_ADMIN, alta de dueños, onboarding del dueño, permisos finos y selector de contexto.
Fase 2: inmuebles, unidades, contratos, expedientes y vistas OWNER/PROPERTY_ADMIN.
Fase 3: cobranza, MonthlyLedger, convenios, MonthlySummary, exportaciones PDF/Excel y portal ACCOUNTANT.
Fase 4: portal TENANT, tickets, desocupación, mantenimiento y vacancias operadas por dueño/admin.
Fase 5: endurecimiento de seguridad, webhooks/adapters reales, performance, pruebas E2E y salida a producción.
Fase 2 posterior: portales completos de REAL_ESTATE_AGENT y MAINTENANCE_PROVIDER.
Plan de Pruebas
Seguridad: login, MFA, refresh rotation, blacklist de tokens, selector de contexto, rate limiting, headers y auditoría.
Autorización: matriz completa por rol; ACCOUNTANT multi-owner con acceso total al owner actual y sin fuga entre dueños.
Reportes: cálculo correcto de MonthlyLedger y MonthlySummary, vista caja + compromisos, exportación PDF/Excel y consistencia entre dashboard, resumen y facturación.
Cobranza: pagos parciales, pagos por Mercado Pago, SPEI manual, convenios, saldos pendientes y reflejo en reportes.
Operación: tickets, presupuestos, egresos comprometidos vs pagados, vacancias y overrides por inmueble.
E2E v1: SUPER_ADMIN -> OWNER onboarding -> alta inmueble/unidad/contrato -> TENANT paga -> OWNER valida -> ACCOUNTANT descarga resumen y reporte de facturación.
Supuestos y Defaults
El workspace actual está vacío, así que no hay restricciones heredadas de implementación previa.
ACCOUNTANT es multi-owner y siempre entra a un contexto de dueño; dentro de ese contexto ve todos los inmuebles del owner.
La base contable del resumen en v1 será caja + compromisos, no devengado puro.
Mercado Pago será la única pasarela real en v1; WhatsApp y n8n quedan listos por adapters/eventos para activación progresiva.
Los borrados sensibles serán soft delete salvo políticas futuras explícitas de purga.

## Validaciones de token en CADA request

```java
// JwtAuthenticationFilter.java — se ejecuta en CADA request
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest req, ...) {
        
        String token = extractToken(req);
        if (token == null) { chain.doFilter(req, res); return; }

        // 1. Verificar firma RS256
        Claims claims = jwtParser.parseSignedClaims(token).getPayload();

        // 2. Verificar expiración (automático con jjwt)

        // 3. Verificar que el jti NO está en la blacklist de Redis
        if (revokedTokens.exists("revoked:" + claims.getId())) {
            throw new TokenRevokedException();
        }

        // 4. Verificar tipo de token
        String tokenType = claims.get("type", String.class);
        String requestPath = req.getRequestURI();

        if ("BASE".equals(tokenType)) {
            // Token base SOLO puede acceder a:
            // - GET /api/auth/contexts
            // - POST /api/auth/select-context
            if (!isContextSelectionEndpoint(requestPath)) {
                throw new ContextRequiredException();
            }
        }

        // 5. Establecer contexto del tenant para RLS
        String ownerId = claims.get("ownerId", String.class);
        if (ownerId != null) {
            TenantContext.setCurrentOwner(UUID.fromString(ownerId));
        }

        // 6. Establecer autenticación en SecurityContext
        var auth = new JwtAuthentication(claims);
        SecurityContextHolder.getContext().setAuthentication(auth);

        chain.doFilter(req, res);
    }
}
```

// TenantConnectionInterceptor.java — ANTES de cada query SQL
public class TenantConnectionInterceptor implements StatementInspector {
    
    @Override
    public String inspect(String sql) {
        UUID ownerId = TenantContext.getCurrentOwner();
        if (ownerId != null) {
            // Ejecutar SET antes de cada query

            );
        }
        return sql;
    }
}

## Action tokens (links de WhatsApp)

```java
// Validación cuando un proveedor abre un link de WhatsApp
@GetMapping("/actions/preview/{token}")
public ResponseEntity<?> previewAction(@PathVariable String token) {
    ActionToken at = actionTokenService.findByToken(token);
    
    // 1. ¿Existe?
    if (at == null) throw new NotFoundException();
    
    // 2. ¿Expiró?
    if (at.getExpiresAt().isBefore(now())) throw new TokenExpiredException();
    
    // 3. ¿Ya fue usado?
    if (at.getStatus() == ACCEPTED || at.getStatus() == REJECTED)
        throw new TokenAlreadyUsedException();
    
    // 4. Registrar visualización
    at.setViewedAt(now());
    at.setStatus(VIEWED);
    
    // 5. Retornar info mínima SIN datos sensibles
    return ok(ActionPreviewDTO.from(at)); // solo tipo, inmueble, descripción
}

@PostMapping("/actions/respond/{token}")
@PreAuthorize("isAuthenticated()") // DEBE estar logueado
public ResponseEntity<?> respondToAction(
    @PathVariable String token,
    @Valid @RequestBody ActionResponseRequest req,
    @AuthenticationPrincipal JwtUser user,
    HttpServletRequest httpReq
) {
    ActionToken at = actionTokenService.findByToken(token);
    
    // Todas las validaciones anteriores +
    // 6. ¿El usuario autenticado es el destinatario del token?
    if (!at.getRecipientUserId().equals(user.getId()))
        throw new UnauthorizedException("Este link no es para ti");
    
    // 7. Registrar respuesta con auditoría completa
    at.setRespondedAt(now());
    at.setStatus(req.accepted() ? ACCEPTED : REJECTED);
    at.setResponseNotes(req.notes());
    at.setIpAddress(getClientIp(httpReq));
    at.setUserAgent(httpReq.getHeader("User-Agent"));
    
    // 8. Disparar notificaciones al dueño
    notificationService.notifyOwner(at);
}
```

## Validaciones adicionales que aplican a TODO

```
1. CSRF: Deshabilitado para API (stateless JWT), habilitado para webhooks con HMAC.

2. CORS: Solo permitir orígenes de la app:
   - https://app.propgest.mx
   - https://admin.propgest.mx (si separas el admin)
   - http://localhost:5173 (solo en desarrollo)

3. Content-Type: Solo aceptar application/json en la API.
   Rechazar cualquier otro tipo con 415.

4. Tamaño de request:
   - JSON body: max 1MB
   - Upload de fotos: max 5MB por foto
   - Upload de PDF: max 10MB

5. SQL Injection: Spring Data JPA con consultas parametrizadas.
   NUNCA concatenar strings en queries.
   NUNCA usar @Query con string interpolation.

6. XSS: React escapa por defecto. NUNCA usar dangerouslySetInnerHTML.
   CSP header bloquea scripts inline.

7. Audit log: TODA acción de escritura (POST/PUT/DELETE) se loggea:
   - Quién (userId, role)
   - Qué (endpoint, recurso)
   - Desde dónde (IP, User-Agent)
   - Cuándo (timestamp)
   - Contexto (ownerId)

8. Headers de seguridad (en cada response):
   - Strict-Transport-Security
   - X-Content-Type-Options: nosniff
   - X-Frame-Options: DENY
   - Content-Security-Policy
   - Referrer-Policy
   - Permissions-Policy
```
